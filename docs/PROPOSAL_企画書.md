# 企画書 — Priveat（プライベート）

> このドキュメントは「なぜ作るか」「何を作るか（全体像）」を定義する。詳細な実装方針は `SPEC_アプリ仕様書.md`、作業進捗は `PROGRESS_進捗表.md` を参照。

## 1. 概要

スマホ単体（オンデバイス）で動くローカルAIを使った、プライバシー重視・オフライン対応・運用コストほぼゼロの食事／体重管理アプリ。

ユーザーはチャットUIに「食べたもの」をテキスト・写真・スクリーンショットで送るだけで、摂取カロリーと消費カロリーが集計され、AIから食事アドバイスを受けられる。記録は端末内に蓄積され、別画面でいつでも履歴を振り返れる。

**名称**: Priveat（プライベート）。private（端末内で完結する＝プライベート）＋ eat（食べる）の造語。最大の強みである「食事記録が端末の外に出ない」を名前で表現している。読みは "private" と同じ「プライベート」。
> 注: 造語のため重複は少ないと見込まれるが、公開前に App Store / Google Play での名称重複と商標を要確認。

## 2. 背景と課題

開発者自身が現在2つの方法でダイエット管理をしているが、それぞれに不満がある。

- **あすけん（既存の食事管理アプリ）**: 食べたものを毎回手入力するのが面倒。消費カロリーの入力も手間。
- **ChatGPT（代替として使用中）**: 食べたものを列挙すれば摂取カロリーを概算してくれ、フィットネスアプリのスクショを送れば消費カロリーも読み取ってくれる。体重もスクショで把握してくれる。しかし**ダイエット管理専用ではないため、過去の記録をさかのぼれない**。

この2つの不満は、それぞれ別のUIで解決すべきもの:

- 「入力が面倒」→ チャット＋AIによる低摩擦なキャプチャで解決
- 「履歴をさかのぼれない」→ 構造化された記録とダッシュボード／履歴画面で解決

## 3. 解決アプローチ（核心の設計思想）

**「LLMには言葉とコーチングを、数字はOCRと食品DBに」** という役割分担を徹底する。

オンデバイスの小型LLMはクラウドのGPTより栄養知識が弱く、カロリー数値を直接出させると精度が落ちる。そこで:

- LLM/VLM は「入力（テキスト・写真）の意味を理解する」ことと「アドバイス・履歴への質問応答」に専念する
- 数値（カロリー・栄養素・体重）は、できる限り OCR（文字の直接読み取り）と食品DB（公式の成分表）から得る

これにより、小型のオンデバイスLLMでも実用的な精度を確保する。

## 4. ターゲットユーザー

- 第一ユーザーは開発者自身（自分用）。まずは自分が毎日使える品質を目標にする。
- 想定像: ゆるやかな減量・健康維持をしたい人。手入力の面倒さで挫折した経験がある人。プライバシーやオフライン動作を重視する人。

## 5. 提供価値

- **入力が速い**: チャットに送るだけ。テキスト／料理写真／食品ラベル／フィットネスのスクショに対応。
- **振り返れる**: ChatGPTにない、日別・週別の履歴とグラフ。
- **プライベート**: 健康データが端末から出ない。オフラインで動く。
- **無料で使い続けられる**: クラウドAPI課金・サーバー費が発生しない。

## 6. スコープ（MVP）

MVPで実現する範囲:

- チャットUIでの食事入力（テキスト）→ LLMで構造化 → 食品DBで摂取カロリー集計
- 食品ラベル写真・フィットネスアプリのスクショ → OCRで数値読み取り
- 料理写真 → VLMで料理名推定＋ユーザーによる量の確認（VLMは後回しでも可）
- 体重・消費カロリーの記録（OCR またはヘルスデータ連携、または手入力）
- 端末内DB（SQLite）への記録保存
- ホーム画面（今日のサマリー）と履歴画面（日別・週別の振り返り）
- LLMによる簡単なアドバイス／コーチング

## 7. 非スコープ（当面やらないこと）

- 複数端末間のデータ同期（将来、必要になったらクラウド同期を別途検討）
- ソーシャル機能・他ユーザーとの共有
- 課金・サブスクリプション
- 医療目的の診断・治療補助（あくまで自己管理ツール）
- 厳密な栄養指導（管理栄養士の代替ではない）

## 8. 成功基準

- 開発者自身が、あすけん／ChatGPTでの現運用を完全に置き換えて毎日使い続けられる。
- 食事入力が「チャットに一文送る」だけで完了し、手入力の面倒さがない。
- 「先週より食べ過ぎ？」のような履歴ベースの問いに、アプリ内で答えられる。
- オフライン・追加課金ゼロで上記が成立する。

## 9. 制約と前提

- **プラットフォーム**: React Native + Expo（iOS / Android 両対応）
- **必須**: React Native の New Architecture。Expo Go は使わず development build（dev client）前提。
- **AI**: クラウドではなくオンデバイス（ローカル）AIを使用。
- **端末リソースとのトレードオフ**: 運用コストを「お金」から「端末のRAM・バッテリー・ストレージ（モデル1〜4GB）」と「クラウドGPT比での精度低下」に移す。精度低下は OCR＋食品DB で補う。
- **Android のばらつき**: iOSはMetalで快適。AndroidはGPU/NPU高速化が端末依存でCPUフォールバックも多い → 端末性能に応じてモデルサイズを切り替える方針。

## 10. コスト

- **変動費（運用コスト）**: ほぼ0。推論・OCR・食品DB照合はすべて端末内で完結。モデル配布はHugging Faceの無料ホスティング＋初回ダウンロード方式。
- **固定費（ストア公開時のみ）**: Apple Developer Program 99ドル/年、Google Play 登録料 25ドル（一度きり）。自分用に dev client で使うだけなら不要な場合もある。
  - ※金額は2026年6月時点。公開前に最新を要確認。

## 11. リスクと対策

- **小型LLMの精度不足** → 数値はOCR＋食品DBに任せ、LLMは言葉の理解に限定。量はユーザー確認。
- **料理写真からのカロリー推定が不正確（量の推定は原理的に困難）** → VLMは料理名の識別までに留め、量はユーザーがタップで確認・調整。MVPでは後回し可。
- **Androidの性能ばらつき** → 端末性能に応じたモデルサイズ切替。低スペック端末はテキスト中心の軽量モデル。
- **ボイラープレートが New Architecture 非前提・更新が古い** → 骨組みとして使い、依存を自分で更新する前提。
- **日本食品DBのカバレッジ（市販品・外食が成分表にない）** → 市販品はラベルOCR＋キャッシュ、外食は概算テーブル＋ユーザー登録で補完。
